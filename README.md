# Менеджер резервных копий через Telegram

## Зачем это нужно

Программа помогает **сохранять копии ваших файлов** не на отдельном облачном диске с подпиской, а **в вашем собственном чате в Telegram** — в отдельной теме группы, куда имеете доступ только вы и те, кого вы туда добавили. Так вы можете **хранить большие объёмы данных** (в разумных пределах правил мессенджера) и **при необходимости вернуть файлы обратно** на компьютер.

Это не «ещё один мессенджер», а **инструмент для бэкапа**: вы указываете, что нужно отправить, программа сама готовит данные и перекладывает их в выбранное место в Telegram, ведёт учёт и позволяет **восстановить** то, что уже было отправлено.

## Роль **7-Zip** в приложении

Программа **не пишет свой формат архива** — она вызывает установленную у вас утилиту командной строки **7-Zip** (`7z`). Это стандартный инструмент сжатия и шифрования.

**Что именно делает 7-Zip по заданию программы:**

- Собирает выбранные файлы и папки в архив **формата 7z** с **шифрованием** (пароль задаёте вы на сессию бэкапа).
- **Скрывает имена файлов внутри архива** вместе с содержимым — по одному только списку частей снаружи нельзя понять структуру каталога.
- **Делит архив на несколько файлов-частей** фиксированного размера (в продукте — **до 1999 МБ на часть**): так каждая часть укладывается в практичный лимит Telegram; граница «около 2 ГБ» у мессенджера ненадёжна, поэтому заложен **запас**.
- Для **восстановления** на другом компьютере те же части скачиваются и **снова обрабатываются через 7-Zip** (или совместимый распаковщик) с тем же паролем.

Итого: **7-Zip = упаковка, шифрование и нарезка**; **программа = очередь, вызов 7-Zip, отправка частей в Telegram, учёт и восстановление**.

## Что умеет программа

- **Резервное копирование** — берёт файлы и папки из **рабочей папки (inbox)** или с **прямых путей на диске**, передаёт их в 7-Zip, затем отправляет получившиеся **части архива** в выбранную **тему** группы в Telegram.
- **Очередь и конвейер** — в каждый момент **полностью обрабатывается один логический объект** (один набор к бэкапу): архив создаётся → все его части отправляются → **временные файлы архива с диска удаляются** → только потом начинается следующий объект. Так не копятся несколько огромных черновиков одновременно.
- **Учёт состояния** — на компьютере ведётся **локальная база данных**: какие части в очереди, какие отправлены, какие сбойнули, какие сообщения в Telegram им соответствуют. Это нужно для **возобновления после обрыва или перезагрузки** и для **restore**.
- **Проверка передачи** — логика приложения сама сверяет успешность шагов (ответы Telegram + согласованность записей в базе); при временных ошибках **повторяет попытки** с ограничением числа попыток; по итогам сессии выводит **список проблемных исходных путей**.
- **Журнал в текстовом файле** — дублирует смысл статусов для человека (удобно открыть блокнотом); источник правды для автоматики — база.
- **После успешной отправки (режим inbox)** исходники **переносятся** в соседнюю папку (`move`, не копирование — объём на диске от этого не удваивается). **Окончательное удаление** из этой папки — только через отдельный экран программы и **двойное подтверждение**. В режиме **прямых путей** файлы на диске не переезжают.
- **Восстановление** — по записям в базе запрашиваются части из Telegram, кладутся на диск, дальше вы распаковываете их через 7-Zip с паролем; предусмотрены **докачка** и статусы при обрывах.
- **Интерфейс в терминале (TUI)** — сессии, пароль, тема, прогресс, оценка времени, стоп/тишина, сводка ошибок.

## Как устроен поток данных (логика кратко)

1. **Вход:** стабильные файлы попадают в очередь (скан рабочей папки и/или слежение за изменениями).
2. **Упаковка:** для следующего элемента очереди вызывается **7-Zip** → в **рабочий кеш** пишутся части `.7z.001`, `.7z.002`, …
3. **Отправка:** каждая часть уходит в Telegram как **документ** в нужную тему; после **успешного** ответа фиксируется запись в базе (связь «файл на диске ↔ сообщение в чате»).
4. **Очистка кеша:** когда **все** части **этого** объекта отправлены — временные файлы из рабочего кеша **удаляются**.
5. **Пост-обработка:** для inbox — перенос исходников в папку «перенесённые»; для прямых путей — шаг пропускается.
6. **Завершение сессии:** когда очередь отработана по выбранным правилам (ручной стоп и/или пауза без новых файлов), в тему уходит **сводное текстовое сообщение** (шаблон настраивается).

При сбое на шаге 3 часть может остаться в состоянии «отправляется» — после перезапуска программа по базе **повторит** недостающее, не начиная архив заново с нуля (если временные файлы ещё есть).

## Что вам понадобится

- **Аккаунт Telegram**, **группа с темами** и **бот** с правами отправлять файлы и сообщения в темы. Настройка бота — через стандартные средства Telegram.
- **Установленный 7-Zip** в среде, где запускается программа (в типичной установке он уже входит в образ/пакет вместе с приложением — см. руководство по развёртыванию).
- **Компьютер**, на котором крутятся программа и сопутствующие службы, пока идёт бэкап или восстановление.
- **Стабильный интернет**; при обрывах — повторы и возобновление по базе.
- **Свободное место на диске:** исходные данные + **пик на одну текущую нарезку** в кешe; постоянный «второй полный архив» программа не оставляет по завершении объекта.
- **Развёртывание окружения** — обычно это **несколько компонентов на одной машине** (хранилище учёта, служба для приёма **больших** файлов в Telegram, сам клиент с интерфейсом). Конкретные шаги — в **[IMPLEMENTATION_GUIDE.md](IMPLEMENTATION_GUIDE.md)**.

## Важно понимать

- Резервная копия живёт **в Telegram** — соблюдайте **правила сервиса** и оценивайте риски (лимиты, блокировки, политика использования).
- **Полную историю чата** «как в обычном клиенте» программа не обходит: в учёт попадает в основном то, что **отправил бот через это приложение** (и то, что вы явно добавите вручную в учёт, если такая функция появится).
- **Пароль от архива** храните отдельно; без него распаковать 7z-архив нельзя.
