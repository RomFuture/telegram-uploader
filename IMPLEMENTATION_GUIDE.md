# Руководство по реализации: Telegram Uploader (TUI)

Документ задаёт **порядок работ** и **критерии готовности** этапов. Детальная архитектура, обоснования и ограничения — в основном плане проекта **«Telegram uploader TUI»** (файл плана в Cursor: `telegram_uploader_cli_7bb2b57f.plan.md`).

---

## Организация кода: Onion Architecture и SOLID

Проект следует **луковичной (onion) архитектуре**: зависимости направлены **от периферии к центру**; **ядро** не знает о деталях БД, HTTP, Docker и TUI.

| Слой | Содержание | Правило |
|------|------------|---------|
| **Domain** (центр) | Сущности, value objects, инварианты (статусы томов, правила очереди), доменные ошибки | **Без** импортов PostgreSQL, `requests`, Textual, путей к `7z` |
| **Application** | Use cases / сервисы приложения: «забэкапить объект», «завершить сессию», «восстановить том»; **порты** — абстрактные интерфейсы к хранилищу, часам, отправке файлов | Зависит только от domain + портов (абстракций) |
| **Infrastructure** | Реализации портов: SQL-репозитории, HTTP-клиент к локальному Bot API, ФС, вызов `7z`, TXT-журнал, `inotify` | Может зависеть от domain + application-контрактов |
| **Presentation (TUI)** | Экраны, команды, привязка событий к use cases | Только вызывает application-слой; **без** бизнес-логики переноса/ретраев в виджетах |

**Composition root** (точка сборки): один модуль/пакет, где создаются реализации и внедряются в use cases (при необходимости — простой DI без тяжёлого фреймворка).

**SOLID (практически):**

- **S** — один модуль/класс — одна зона ответственности (например, «клиент Bot API» ≠ «политика ретраев» ≠ «рендер TUI»).
- **O** — новое поведение через новые реализации **портов** или расширение use cases, не правки ядра ради «ещё одного хранилища».
- **L** — реализации портов взаимозаменяемы в тестах (фейки/in-memory).
- **I** — узкие интерфейсы: отдельно «отправить документ», «прочитать тома сессии», а не один «God-интерфейс».
- **D** — use cases и domain зависят от **абстракций** (протоколы/ABC в Python), конкретика подключается снаружи.

Структура каталогов (пример, уточняется при старте): `domain/` → `application/` → `infrastructure/` → `presentation/` (TUI), плюс `composition/` или `main.py` для сборки.

## Негласные правила (везде в коде)

1. **Транспорт:** Bot API через **локальный** [`telegram-bot-api`](https://github.com/tdlib/telegram-bot-api); тома **1999 МБ** (`-v1999m`), не «до 2 ГБ».
2. **Очередь:** один **`source_item`** в конвейере 7z→TG до конца; следующий не архивировать, пока текущий не завершён.
3. **Рабочий кеш 7z:** после успешной отправки **всех** томов объекта — удалить временные файлы **до** следующего объекта.
4. **Inbox:** после успеха — `move` в соседнюю папку; **прямые пути** — без `move`, `moved_to_path` = NULL.
5. **Удаление перенесённых из соседней папки** — только TUI, **двойное согласие**; автоудаления нет.
6. **Секреты** не в git; пароли/токены не в логах и TXT-журнале.
7. **История чата целиком** через Bot API недоступна — не обещать полный sync; наследие — ручной импорт / roadmap MTProto.

## Этап 1. Docker Compose: инфраструктура

- [ ] `docker-compose.yml`: сервисы **`postgres`**, **`telegram-bot-api`**, **`app`**.
- [ ] Образ `app`: Python, **`7z`** в контейнере, зависимости приложения.
- [ ] Тома: путь к **«ТГ-диску»**, **рабочему кешу томов**, **папке перенесённых** (смонтировать с хоста).
- [ ] Переменные: `BOT_TOKEN`, `TELEGRAM_BOT_API_URL`, `DATABASE_URL`; для сервера Bot API — `TELEGRAM_API_ID`, `TELEGRAM_API_HASH` ([my.telegram.org](https://my.telegram.org/apps)).
- [ ] Healthcheck для PostgreSQL; `depends_on` / ожидание готовности БД перед миграциями.
- [ ] `.env.example` без реальных секретов.

**Готово, когда:** `docker compose up` поднимает три сервиса; `docker compose run --rm app` выполняет smoke (например `getMe` к локальному Bot API) и подключается к БД.

- [ ] Загрузка из env + опционально файла; валидация обязательных полей.
- [ ] Пути: ТГ-диск, папка перенесённых, рабочий кеш 7z, TXT-журнал.

## Этап 4. Клиент Telegram (локальный Bot API)

- [ ] Обертка над HTTP: базовый URL из конфига, не `api.telegram.org`.
- [ ] Методы v1: `getMe`, `sendDocument` (с `message_thread_id`), `sendMessage`, при необходимости `createForumTopic`.
- [ ] Обработка **FloodWait** (`retry_after`), экспоненциальный бэкофф, лимит частоты отправок (настраиваемо).

**Готово, когда:** из скрипта/CLI отправляется один тестовый документ в выбранную тему супергруппы.

## Этап 5. Модуль 7-Zip

- [ ] Вызов `7z` с `-t7z`, `-p`, `-mhe=on`, `-v1999m`; корректные коды возврата и разбор ошибок.
- [ ] Рабочий каталог для томов текущего объекта; после успешной отправки всех томов — **удаление** файлов этого набора из кеша.
- [ ] Поддержка файлов и каталогов; не-ASCII пути с учётом ОС.

**Готово, когда:** локально создаётся зашифрованный split-архив и затем каталог кеша очищается сценарием «успех».

---

## Этап 6. Worker: один объект end-to-end

- [ ] Для одного `source_item`: 7z → запись томов в БД → по очереди `sendDocument` → после каждого успеха транзакция `sent` + `file_id`.
- [ ] После последнего тома: очистка рабочего кеша; для inbox — `move` в соседнюю папку; для прямых путей — без move.
- [ ] Ретраи до **N** на том/объект; по исчерпании — статус ошибки и запись в TXT-журнал.

**Готово, когда:** один реальный файл/каталог проходит путь до Telegram и отражается в БД; после прогона нет лишних `.7z` в рабочем кеше.

---

## Этап 7. Очередь, сканер, вотчер

- [ ] Периодический скан «ТГ-диска» и/или `inotify` (debounce); стабилизация размера файла до постановки в очередь.
- [ ] `INSERT` в `source_items` со статусом `queued`; дедуп по пути + размер/mtime.
- [ ] Один worker на активную сессию; новые элементы не сбрасывают текущую работу.

**Готово, когда:** докидывание файлов в папку во время сессии добавляет задачи без перезапуска worker.

## Этап 8. Сессии, политика завершения, сообщение в тему

- [ ] Модель `upload_sessions`; привязка `source_items` к сессии.
- [ ] Завершение: ручной стоп и/или «тишина **T** секунд» после пустой очереди.
- [ ] По завершении: `sendMessage` в тему по шаблону из конфига.
- [ ] TXT-журнал: дописывание статусов; автопроверка успеха по БД и API (без обязательной ручной сверки журнала).

**Готово, когда:** полная сессия с несколькими объектами завершается сообщением в TG и сводкой в TUI.

## Тестирование

Тесты — часть **Definition of Done** для каждого значимого изменения; приоритет — **быстрые** unit-тесты и **изолированные** интеграционные, без обязательной живой Telegram на каждый commit.

### Уровни

| Уровень | Назначение | Типичное содержание |
|---------|------------|---------------------|
| **Unit** | Доменные правила, политики (ретраи, переходы статусов), парсинг конфига, чистые функции | Без сети и БД; моки **портов** и времени (`clock`) |
| **Интеграционные** | Реализации репозиториев, миграции, обёртка БД | Реальный **PostgreSQL** в CI: сервис job, `testcontainers`, или `docker compose` для тестов |
| **Контрактные / HTTP** | Клиент Bot API | Подмена ответов (`respx`, `responses`, `httpx` MockTransport) или записанные JSON-ответы; проверка **FloodWait** и повторов |
| **Интеграция с 7z** | Сборка/разбор томов | Либо реальный `7z` в job-образе (как в prod), либо скрипт-заглушка с тем же контрактом CLI |
| **E2E / smoke** | Полный путь в тестовой среде | По желанию: не в каждом PR; ручной или ночной прогон с тестовым ботом и группой |

### Принципы (связь с SOLID)

- **Порты** в application-слое позволяют подставлять **фейки** в unit-тестах без БД и Telegram (**D**, **L**).
- **Тонкие интерфейсы** упрощают моки и меньше ломаются при рефакторинге (**I**).
- Тесты **не** должны импортировать TUI из тестов domain — соблюдать границы слоёв.

### CI

- [ ] В pipeline: `pytest` (unit + integration по маркерам), линтер, форматтер (check).
- [ ] Секреты в CI **не** хранить в логах; для интеграций с БД — переменные сервиса или ephemeral Postgres.
- [ ] Порог покрытия: задать **минимальный** (например, только `domain` + `application`) или ориентир по процентам — по договорённости команды; главное — **критические пути** (worker, транзакции `sent`, ретраи) под тестами.

### Чеклист по этапам (когда писать тесты)

- **Этапы 2–3:** миграции «поднимают схему»; тесты репозиториев на чистой БД.
- **Этапы 4–5:** HTTP-клиент и 7z — отдельные тесты с моками/реальным бинарником в образе.
- **Этап 6:** сценарии worker с подменёнными портами (успех, сбой после N попыток, частичный `sending`).
- **Этапы 9–10:** UI-тесты TUI — опционально; приоритет — тесты use cases без Textual.
