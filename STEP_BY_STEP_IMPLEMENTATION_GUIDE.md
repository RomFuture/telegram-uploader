# Step-by-Step Project Creation Guide

This guide describes how to build the project from scratch, in strict sequence, with implementation checkpoints.

Sections **4–8** below include an **Implementation status** block: what is already in the repo versus what remains to wire end-to-end (use cases, DB repositories, GUI).

## 0. Project Goal and Constraints

Build a Linux desktop application that stores files through messenger providers.

- `v1` real provider: Telegram only.
- Architecture: provider-agnostic core (`StorageProviderPort`), Telegram as first adapter.
- Data flow target:
  - `Linux GUI -> App Backend Receiver -> ApplicationUseCases -> Celery Queue/Worker -> PostgreSQL -> Messenger`.
- Grouping logic is managed in the app GUI, not via Telegram topics.
- Parallel pipeline is required: archiving, upload, cleanup in overlap for different source items.

## 1. Mandatory Product Rules (define before coding)

Write these rules in your internal spec first and treat them as non-negotiable:

1. Interface language policy:
   - All user-facing interface text must be in English.
   - This includes: GUI labels, buttons, tooltips, dialogs, validation errors, notifications, and logs visible in UI.
2. Telegram usage:
   - Send files into one target Telegram group (no topics, no `message_thread_id`).
3. Security:
   - Archive encryption is mandatory.
   - Encryption key is auto-generated by app by default.
   - User can override/change the key in settings.
4. Restore metadata:
   - After upload, save messenger identifiers and download reference in DB (`external_file_id`, `external_message_id`, `provider_download_ref`).
5. Filename policy:
   - Telegram uploaded volume names must include a hashed user filename component.
6. Display name:
   - Original user-facing file name (e.g. `Мысли.txt`) must be persisted as `display_name` in `source_items` table at the moment of enqueue.
   - GUI always shows `display_name`, never derives it from `source_path` at render time.
7. Source files:
   - App does not auto-move original files to a service folder.
   - User controls data organization through GUI profile/workspace.

3. Add indexes:

**Implementation status**

1. Implement `TelegramProviderV1` as `StorageProviderPort` implementation.

3. Implement error normalization:

4. Ensure upload response mapping fills:
   - `external_file_id`
   - `external_message_id`
   - `provider_download_ref`
5. Add provider-level upload filename builder:
   - Include hashed user filename component in outgoing volume names.

**Implementation status**

- `src/infrastructure/providers/telegram_provider.py`: `TelegramProviderV1` implements `StorageProviderPort` (HTTP to local Bot API URL).
- `upload_file` uses **`multipart/form-data`** with a real file body (`sendDocument` + `document` + `chat_id`) — suitable for local Bot API and production.
- Error classification and limits are implemented; **live upload to a real group** still needs integration testing with a real bot token and `chat_id`.

Checkpoint:
- Test file upload returns and persists all required restore metadata.

## 7. 7-Zip and Encryption Pipeline

1. Integrate `7z` invocation with required options:
   - encrypted archive
   - hidden metadata where applicable
   - volume splitting size for Telegram constraints
2. Use direct binary upload policy:
   - Upload encrypted split `7z` volumes to Telegram as binary files (`sendDocument`) without Base64 conversion.
3. Implement key lifecycle:
   - Auto-generate key when session starts if user did not provide one.
   - Allow user-provided key override in settings.
4. Implement deterministic naming strategy for archive volumes:
   - Hash must be computed from `display_name` (captured in DB at enqueue time), not from `source_path.name` at archive time.
   - Reason: file on disk may be renamed between enqueue and archive; `display_name` is immutable after enqueue.
   - include part numbering
5. Ensure temp artifacts are scoped and traceable to source item.
6. Cache directory layout (rooted at `ARCHIVE_CACHE_DIR`, default `/tmp/telegram_uploader`):

```
/tmp/telegram_uploader/
  <source_item_id>/          ← one folder per source item (UUID)
    raw/                     ← volumes as created by 7z; each volume deleted immediately after copy to outgoing/
    outgoing/                ← volumes renamed with hashed names ready for upload
    volume_manifest.json     ← ordered part map (raw name ↔ outgoing name)
```

7. `raw/` volume deletion policy:
   - Each raw volume is deleted immediately after it is copied to `outgoing/`.
   - This keeps peak disk usage at: original file + full `outgoing/` + one raw volume at a time.
   - Do not wait until cleanup stage to delete raw volumes.

**Implementation status**

- `src/infrastructure/archive/seven_zip_service.py`: `SevenZipService.archive(source_path, output_dir, display_name, ...)` runs `7z` with `-t7z -mhe=on -v1999m`, produces volumes in `raw/`, copies each to `outgoing/` with hashed name, then immediately deletes the raw volume. Writes `volume_manifest.json`.
- `ArchiveVolume` carries `outgoing_path`, `outgoing_file_name` (no `raw_path` — raw file is deleted after copy).
- `build_hashed_volume_name(display_name, part_number)` hashes `display_name` (not `source_path.name`).
- `AppConfig.archive_cache_dir` reads `ARCHIVE_CACHE_DIR` from environment (default `/tmp/telegram_uploader`).
- Docker image installs **`p7zip-full`** so `7z` is available in app and worker containers.

Checkpoint:
- Local run produces encrypted split volumes with expected names.

## 8. Celery Orchestration (Parallel Stages)

1. Define task pipeline stages:
   - Prepare source
   - Archive source
   - Upload volume(s)
   - Persist provider metadata
   - Cleanup temp cache
2. Implement overlapping execution policy:
   - Source B archiving may start while source A uploads.
   - Cleanup must run only after successful upload confirmation for that source.
3. Add retries and backoff:
   - per volume
   - bounded retry policy
4. Add idempotency keys/guards to avoid duplicate DB writes.

**Queue and worker topology (current default)**

- Three Redis queues: `archive`, `upload`, `cleanup`.
- **Four worker processes** in `docker-compose.yml`:
  - **2** workers consume only `archive` (CPU/disk-heavy stage),
  - **1** worker consumes only `upload` (network-heavy),
  - **1** worker consumes only `cleanup` (fast local I/O).
- Code: `src/infrastructure/worker/celery_app.py` (broker/backend from `REDIS_*`, `task_routes`), `src/infrastructure/worker/tasks.py` (stub tasks: `archive_volume`, `upload_volume`, `cleanup_volume`).
- Next: enqueue from use cases, chain tasks per volume, persist IDs in DB, configure retries/backoff and idempotency.

Checkpoint:
- Multiple source items process in overlap.
- Statuses remain consistent under retries and partial failures.

## 11. Restore Workflow

1. Query DB for uploaded artifacts by session/item.
2. Use `provider_download_ref` (plus provider IDs) to retrieve files.
3. Support resume/doretry for interrupted downloads.
4. Reassemble downloaded archive volumes in deterministic order.
5. Decrypt and extract archive with chosen key.
6. Provide clear UI status for restore progress/errors.

Checkpoint:
- Restore works after app restart and network interruption.

## 12. Testing Strategy

1. Unit tests:
   - domain rules
   - use-case orchestration
   - filename hashing policy
   - Celery queue routing (`archive` / `upload` / `cleanup`)
2. Integration tests:
   - DB migrations
   - repository behavior
   - Celery task state transitions (extend beyond route/unit checks once tasks call real services)
3. Provider contract tests:
   - `StorageProviderPort` compliance checks
4. Telegram-specific tests:
   - floodwait/rate limits
   - invalid token
   - no permission in target group
5. E2E smoke:
   - backup and restore from GUI flow

Checkpoint:
- Critical pipeline and restore paths are covered by automated tests.

## 14. Observability and Operational Safety

1. Store operational logs locally.
2. Never send verbose operational text logs into messenger.
3. Add structured events for:
   - task start/finish/retry/failure
   - metadata persistence success/failure
   - restore milestones
4. Redact secrets from logs.

Checkpoint:
- Failures are diagnosable from local logs and DB state.

1. Freeze scope:

2. Run release checklist:

- `v1` is stable, documented, and prepared for future providers (`Max`, `VK`) without rewriting core architecture.
